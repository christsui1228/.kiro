# Vibe 模式开发规范

## Vibe 模式特点
- **快速迭代**: 优先快速实现和验证想法
- **创意优先**: 鼓励创新和实验性方案
- **灵活应用**: 规范作为指导而非严格约束

## 核心规则适用
即使在 Vibe 模式下，仍需遵循：

### 必须遵循的规则
- ✅ **简体中文交流**: 所有对话和注释使用简体中文
- ✅ **现代化技术**: 优先使用最新稳定技术
- ✅ **根治性方案**: 避免临时补丁式解决方案
- ✅ **技术栈一致性**: 遵循项目技术栈选择

### 可以放松的规则
- 🔄 **完整文档**: 可以简化文档要求
- 🔄 **严格流程**: 可以跳过部分设计阶段
- 🔄 **测试覆盖**: 可以延后测试编写

## Vibe 模式交互风格

### 快速响应模式
```
用户: "帮我快速实现一个用户登录"
AI: "基于你的技术栈，我直接提供 FastAPI + SQLModel 的现代化实现..."
```

### 创意探索模式
```
用户: "有什么更好的方式处理文件上传？"
AI: "考虑几种现代化方案：
1. 直传 R2 + 预签名 URL (推荐)
2. 流式上传 + TaskIQ 异步处理
3. 分片上传 + 断点续传..."
```

### 问题解决模式
```
用户: "这个性能问题怎么解决？"
AI: "从根本上解决：
1. 架构层面：引入 NATS 事件驱动
2. 数据层面：使用 Polars + ConnectorX
3. 缓存层面：利用 R2 边缘缓存..."
```

## 快速行动指南

### 代码生成
- 直接生成符合规范的代码
- 包含简体中文注释
- 使用现代化模式

### 问题诊断
- 快速定位根本原因
- 提供系统性解决方案
- 避免临时修补

### 技术选型
- 自动应用技术栈规范
- 推荐最新最佳实践
- 考虑长期维护性
